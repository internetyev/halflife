## 2026-05-09

- **D-006** Version pins for the L1.1 scaffold: Next.js 15.1, React 19, TypeScript 5.7, Tailwind v4 (with `@tailwindcss/postcss`), shadcn/ui (`new-york` style, `neutral` base, RSC + tsx). Tailwind v4 picked over v3 because it's stable as of mid-2025 and removes the JIT/postcss boilerplate; shadcn/ui supports it. Anthropic SDK pinned at `^0.40.0`; `@vercel/kv` and `@vercel/og` included now so L2.3/L2.6 don't reopen this manifest. No `next.config.ts` yet — that lands with L2.1 when the app skeleton goes in.
- **D-007** TypeScript strict-mode dial-up: in addition to `strict: true`, L1.2's `tsconfig.json` enables `noUncheckedIndexedAccess`, `noImplicitOverride`, and `noFallthroughCasesInSwitch`. Cheap insurance against the two failure modes most likely on this codebase — silent `undefined` from cache hits keyed by slug, and a switch on `confidence` levels in the result renderer. Module resolution set to `bundler` to match Next.js 15 / TS 5.7 defaults; `@/*` path alias points at the repo root so imports stay flat (no `src/` directory planned).
- **D-008** Scoring rubric (`docs/methodology.md` v1): six dimensions on 0–10 — task automatability (0.30), tool maturity (0.20), adoption velocity (0.15), HITL necessity (0.15), differentiation moat (0.10), labor-market elasticity (0.10). Score = round(weighted sum × 10). `countdown_years` is a deterministic banded function of score (0–19 → 0.5–2y, 20–39 → 2–4y, 40–59 → 4–7y, 60–79 → 7–12y, 80–100 → 12–20y) with ±5% slug-hash jitter so two equal scores don't render identical countdowns. Banding is generous on the high end and tight on the low end on purpose — a 30-year-old role getting a 1-year countdown kills credibility, and a sub-20 score should feel urgent. `methodology_version` ships in the result JSON and the cache key so a weight change invalidates the cache without a manual purge.
- **D-009** Prompt contract (L1.4, `prompts/role-analysis.md` v1): the `submit_role_analysis` tool returns the **six dimension ratings (score + justification each)**, the named tools, pivot steps, confidence, confidence rationale, sources hints, and the synonym-collapsed `normalized_title` — but **not** `score` or `countdown_years`. Those are computed server-side from the dimensions using the methodology weights and bands, so an internal disagreement between "the model said 72" and "the dimensions average to 51" is impossible by construction. The roadmap leaf L1.4 listed `score`/`countdown_years` in the tool output as shorthand; this decision is the authoritative interpretation. `prompt_version: 1` is added to the cache key alongside `methodology_version` so a prompt edit (system text, schema, examples) invalidates KV without a weight bump.
- **D-010** Tool-use enforcement: server pins `tool_choice: { type: "tool", name: "submit_role_analysis" }` and ignores any non-tool content from the model. The prompt allows internal reasoning but the server discards it — keeps the response surface minimal and prevents accidental free-text leakage into the result card. Dimension `justification` strings and `confidence_rationale` are persisted to KV for evals/debugging but are **not** returned in the public API response; only the user-facing fields cross the API boundary.
- **D-011** L1.5 split into L1.5a (CSV scaffold + procedure, routine-doable) and L1.5b (run the prompt and rate outputs, human-gated by `ANTHROPIC_API_KEY` per `ROUTINE.md` cost-discipline rule that forbids the routine from making live API calls). The 10 baseline roles in `evals/README.md` are chosen to exercise four of the five "Open questions for L1.5" in `prompts/role-analysis.md`: two anchor rows re-run the worked examples (`paralegal`, `kindergarten teacher`) for direct comparison with the in-prompt examples; three are synonym-collapse probes (`attorney → lawyer`, `ux designer → product designer`, `account exec → sales representative`) that double as a cache-hit-rate canary because aliased inputs that fail to normalise will fragment the production KV cache; two are band sanity-checks (`translator` low, `plumber` high); `copywriter` probes the "named, real tools" failure mode and `customer support representative` probes the generic-pivot regression; `radiologist` stresses the calibration line against doom-mongering on a regulated role with a real but slow AI-imaging adoption story. Adversarial / gibberish / fast-moving sub-field probes (the fifth open question) are deferred to a separate confidence-eval rather than mixed into the v1 baseline. CSV columns are flattened to one row per role, with nested fields collapsed into a `raw_json_path` pointer so a human can rate consistency in a spreadsheet without parsing JSON; the full tool output is meant to land alongside the CSV as per-role JSON files referenced by that column.

- **D-016** L2.2 analyze route shape + module split. Three small TS modules behind one route: `lib/scoring/types.ts` (the runtime view of the tool input + the public response shape, with `METHODOLOGY_VERSION` and `PROMPT_VERSION` exported as `as const` for cache-key composition in L2.3); `lib/scoring/index.ts` (`computeScore`, `bandedCountdown` with deterministic FNV-1a slug-hash jitter, `slugify`, `buildResult`); `lib/anthropic/role-analysis.ts` (the v1 system prompt, the tool schema, and the user-message builder, all as TS constants — the route never reads `prompts/role-analysis.md` at runtime). Splitting it this way means L2.3 (KV cache) wraps `analyzeRole`-equivalent logic without touching the route surface, and L3.2 (programmatic seed over 200 roles) can import `buildResult` and the prompt constants from a script without standing up a Next.js server. **Schema inlining decision:** `prompts/role-analysis.md` uses `$ref`/`definitions` for the dimension subschema, but the Anthropic tool-use input_schema validator rejects JSON-Schema `$ref`/`oneOf`/`allOf`/`anyOf` at any level — confirmed today by an MCP tool error reproducing exactly that constraint. The TS schema constant therefore inlines the dimension subschema six times. The two are still semantically identical; the markdown stays the human-readable contract. **Model pinning:** `claude-sonnet-4-6` is hardcoded as a `const` in the route, not read from env. A model bump should be a deliberate code change so the prompt-cache key (which composes `methodology_version × prompt_version`) and the eval baseline both move together; a stray env var should not silently swap the production model. **Caching:** `cache_control: { type: "ephemeral" }` is set on both the system prompt and the tool block — the per-call user message is the only part that varies, so repeat calls inside the 5-minute Anthropic prompt-cache TTL skip re-billing the rubric and schema (which together dominate the prompt token count). **Input bounds:** body must be `{ title: string }`, trimmed to non-empty, ≤200 chars. The cap is generous on title length but bounded enough to keep the prompt-cache key from being abused for cache-stuffing once L2.3 lands. **Boundary discipline:** `dimensions` and `confidence_rationale` are present on the tool input type but deliberately stripped from the API response (D-010); they will be persisted to KV alongside the public fields under L2.3.

## 2026-05-11

- **D-017** L2.3 KV cache layer shape + dual-key write. Cache key composes both versions: `role:m${methodology_version}:p${prompt_version}:${slug}`. Storing both versions in the key (vs. an out-of-band cache-bust step at deploy) means a methodology weight tweak (D-008) or a prompt edit (D-009) silently invalidates every entry on next read with zero ops work — the cost is one extra read/miss per invalidated entry, paid once. TTL is 30 days, per D-005. **Stored payload (`CachedRoleAnalysis`):** the public `RoleAnalysisResult`, plus the model's raw `tool_input` (dimensions + confidence_rationale) for evals/debugging per D-016, plus a `cached_at` ISO string. The boundary discipline from D-010 stands — only `result` crosses the API surface, never the raw tool input. **Synonym handling (dual-key write):** on a fresh miss the route writes the cached entry under both `slugify(rawTitle)` and `slugify(normalized_title)`. Two raw inputs that the prompt collapses to the same canonical role (e.g. `attorney` and `lawyer`) therefore share an entry once either has been seen once. Deliberately *not* shipped: alias-resolution (using the model's normalized_title to find an existing canonical entry *before* the Claude call). The simpler version pays one Claude call per new alias instead of zero, but avoids a second KV read on every miss and removes a write/race window — acceptable until L3.2 surfaces a real cache-fragmentation problem. **Graceful degradation:** when `KV_REST_API_URL` / `KV_REST_API_TOKEN` are absent, the cache module exposes a no-op (every read misses, every write is dropped). This is the dev-laptop story — `next dev` with only `ANTHROPIC_API_KEY` set still serves live results. KV runtime errors are also swallowed, on the principle that a dead cache must never break the analyze endpoint. **Module shape:** `analyzeRole(client, title)` extracted into `lib/anthropic/role-analysis.ts` so the route body is the cache flow only (`lookup → miss → analyzeRole → buildResult → setCachedRole → return`); the L3.2 programmatic-seed script can import `analyzeRole` + `buildResult` directly without standing up Next. The `MODEL` constant moves with `analyzeRole` for the same reason — a model bump and a prompt edit should be one PR. **Cache headers:** `x-halflife-cache: HIT|MISS` on every 200 — handy for browser eyeballing and for the L3.2 script to count Claude calls vs. KV reads.
- **D-018** L2.4 input form scope + result-preview disposability. `app/page.tsx` becomes a `"use client"` component because the form drives a fetch + state machine — making the page itself the client component (vs. introducing a `<RoleForm>` child) keeps L2.4 to a single-file edit and avoids a premature client/server split before L2.5 establishes the real result-card module shape. State is a four-arm discriminated union (`idle | loading | error | result`) rather than separate booleans, so impossible combinations (loading + error, etc.) are unrepresentable. The inline `ResultPreview` renders the raw `RoleAnalysisResult` fields with minimal styling and a tiny `cache HIT/MISS · methodology vN · prompt vN` footer; it is *explicitly disposable* — L2.5 replaces it with the designed card (score gauge, etc.), and the footer goes with it. Submit button stays disabled on empty-trim input and during in-flight; the disabled-state styling uses the existing `--color-primary` tokens from `globals.css` rather than adding new ones, so the shadcn primitives that L2.5 may pull in don't collide with bespoke button styles introduced here. No new dependencies; uses only React, the existing Tailwind v4 tokens, and the `RoleAnalysisResult` type already exported from `@/lib/scoring/types`.
- **D-019** L2.5 result-card module shape, score-band visual taxonomy, no shadcn primitives yet. The polished card lives at `components/result-card.tsx` (server-renderable — no `"use client"`; the parent in `app/page.tsx` already owns the client boundary because it owns the fetch). One file, no sub-components: each section (header / countdown / gauge / confidence / tools / pivot / sources / footer) is small enough that breaking it apart would obscure the read order rather than aid reuse. **Score-band taxonomy** maps the methodology bands (D-008) onto five named visual states — Urgent (0–19, red), At-risk (20–39, orange), Contested (40–59, amber), Durable (60–79, emerald), Stable (80–100, sky) — with each band carrying a one-line blurb the gauge fill colour, and a pill-style chip in the header. Putting the band copy *in* the card module (vs. `lib/scoring/`) is deliberate: it's UI-layer copy that reads with the gauge, and a future per-role static page (L2.7) will likely reuse the same module rather than re-derive the band from raw score in a different file. **Score gauge:** a Tailwind-only `<div role="meter">` with `aria-valuenow={score}` and a coloured fill — no chart library, no SVG. The product needs *one* horizontal bar and a number; bringing in recharts/visx for that would be premature and would land bytes the L4 report page does not yet need. **Confidence handling:** an inline three-state copy map renders the rubric line from `docs/methodology.md`, and `confidence === "low"` adds an amber banner — the methodology already promises a "low-confidence banner and skip share-card generation" treatment, and this card lands the banner half now (the share-card skip is L2.8's concern). **No shadcn primitives:** D-015 reserves shadcn primitives for when a leaf actually needs one. A `<Card>` wrapper around an `<article>` plus six text sections would add a dependency surface and a `cn()` import without changing a pixel — deferred to the first leaf that genuinely needs `<Button>`/`<Dialog>`/`<Tooltip>` interaction. **Sources hint** rendered as small pill chips (verify-against affordance) rather than free text — the prompt returns search terms and publication names, not URLs (D-009), so a pill list reads as "things to google" rather than as a citation list we cannot back. **Normalized-vs-input disclosure:** when the prompt's `normalized_title` differs from the user's trimmed input (case-insensitive), a one-line "you typed X — collapsed to canonical form" hint surfaces under the heading. Cheap user trust win for the synonym-collapse path D-009 promises, and free debugging signal for the L1.5b "did normalisation actually fire?" question.
- **D-020** L2.6 OG share-card route: data source = KV cache (slug-direct lookup), generic fallback on miss, edge runtime, no font subsetting. The route at `app/api/og/[slug]/route.tsx` reads `getCachedRoleBySlug(slug)` against the same KV entries `/api/analyze` writes (D-017's dual-key write means either the raw input slug or the model's `normalized_title` slug resolves), then renders a 1200×630 ImageResponse. **Why KV (vs. query params for score/countdown):** the share URL on a real role page (L2.7) will be a clean `/api/og/paralegal` with no signed params — putting the numbers in the URL would either be tamper-able (anyone could fake a "0.5 years" share image) or require an HMAC the routine doesn't need to invent yet. **Cache-miss fallback** is a generic "How many years until AI replaces your role?" card with the prettified slug as a teaser, not a 404 — share-card crawlers (LinkedIn, Twitter/X, Discord) drop the image silently on non-2xx, so a broken preview is worse than a non-specific one. The route never calls Claude — pre-warming a slug is `/api/analyze`'s job, by design (an OG fetch from a crawler should never trigger a paid API call). **Edge runtime** chosen because `next/og` is built for it, `@vercel/kv`'s REST client is fetch-based and edge-compatible, and crawlers are latency-sensitive. **Visual taxonomy duplicated, not shared:** the band lookup is copy-pasted from `components/result-card.tsx` (D-019) rather than hoisted into `lib/scoring/` because Satori only reads inline styles (no Tailwind classes) — the on-page card uses Tailwind utility classes (`bg-red-500`, etc.), the OG card needs hex (`#ef4444`). One hoist would have to keep two parallel representations anyway; better to keep each module self-contained until a third surface wants the same colours. **CDN headers:** `Cache-Control: public, max-age=3600, s-maxage=86400, stale-while-revalidate=604800`. Share-card crawlers fan out on a long tail (every social-media re-share triggers another fetch days/weeks later), so cheap re-renders dominate; the 30-day KV TTL is the upper bound on staleness anyway. **No custom fonts yet:** Satori falls back to a system font without an explicit `fonts` array. Embedding a brand font would add ~150 KB to the route bundle and require a font file in the repo — deferred until L2.7 stabilizes the brand voice on the per-role page.

- **D-021** L2.7 per-role page data source: precomputed JSON first, KV second, `notFound()` third. `app/role/[slug]/page.tsx` reads `data/roles/<slug>.json` if present (the L3.2 seed of curated top job titles, enabling true SSG with `generateStaticParams` once it lands) and falls back to `getCachedRoleBySlug` so a freshly minted share URL works now, before any seed JSON exists. Raw URL params are re-`slugify`-ed and length-capped at 200 (matching L2.2 input bound) so `/role/Paralegal` resolves to the same entry as `/role/paralegal`. Unknown slugs return a real 404 via `notFound()` — the OG route generic-fallback rule (D-020) is right for crawlers and wrong for HTML pages. `generateMetadata` wires `/api/og/[slug]` (D-020) into the OG + Twitter card tags, so a missing share image is impossible by construction. The page reuses `<ResultCard cache="HIT">` verbatim — the precomputed/KV distinction is invisible to the user. Until L3.2 lands, the file-read is a no-op miss and KV carries every request.
